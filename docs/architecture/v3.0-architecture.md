## 数据流

### 请求处理流程

```
1. 客户端请求
   ↓
2. 负载均衡 (选择健康节点)
   ↓
3. API Gateway
   ├── 认证验证 (JWT/API Key)
   ├── 权限检查
   ├── 限流检查
   └── 请求路由
   ↓
4. 业务逻辑层
   ├── 会话管理 (Redis)
   ├── 数据访问 (分片路由)
   └── 缓存查询
   ↓
5. 存储层
   ├── 分片选择 (按 tenant_id)
   ├── 读写分离
   └── 加密存储
   ↓
6. 响应返回
   ├── 审计日志
   ├── Webhook 触发
   └── 指标收集
```

### 数据分片流程

```
请求 (tenant_id: "tenant-001")
   ↓
ShardingRouter
   ├── 计算: hash("tenant-001") % shard_count
   └── 结果: shard_id = 2
   ↓
Shard 2
   ├── Primary: 写入
   └── Replica: 读取
```

```
┌─────────────────────────────────────────┐
│           uHorse 集群                    │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │ Node 1  │ │ Node 2  │ │ Node 3  │   │
│  │ Active  │ │ Active  │ │ Standby │   │

Node 1 故障:
1. etcd 心跳超时 (30s)
2. 服务自动注销
3. 负载均衡器移除 Node 1
4. Node 3 激活
5. 流量重定向
```

## 部署架构

### Kubernetes 部署

```yaml
# deployments/k8s/uhorse.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: uhorse-gateway
spec:
  replicas: 3
  selector:
    matchLabels:
      app: uhorse-gateway
  template:
    spec:
      containers:
        - name: uhorse
          image: uhorse:3.0.0
          ports:
            - containerPort: 8080
          env:
            - name: ETCD_ENDPOINTS
              value: "etcd:2379"
            - name: REDIS_URL
              value: "redis://redis:6379"
          livenessProbe:
            httpGet:
              path: /health/live
              port: 8080
          readinessProbe:
            httpGet:
              path: /health/ready
              port: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: uhorse-gateway
spec:
  selector:
    app: uhorse-gateway
  ports:
    - port: 80
      targetPort: 8080
  type: LoadBalancer
```

## 扩展性

### 水平扩展

```
扩展触发条件:
- CPU 使用率 > 70%
- 内存使用率 > 80%
- 请求队列长度 > 100
- 响应时间 P99 > 100ms

扩展流程:
1. 监控系统检测到扩展触发条件
2. 通知 Kubernetes HPA
3. 启动新 Pod
4. 新节点向 etcd 注册
5. 健康检查通过
6. 负载均衡器添加新节点
7. 流量开始路由到新节点
```

### 垂直扩展

```
资源配置:
- Small:  2 CPU, 4GB RAM
- Medium: 4 CPU, 8GB RAM
- Large:  8 CPU, 16GB RAM

自动调整:
1. 监控资源使用趋势
2. 预测未来需求
3. 在非高峰期执行升级
4. 滚动更新避免服务中断
```
