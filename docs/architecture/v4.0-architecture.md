# uHorse v4.0 架构说明

本文档只描述 **当前仓库主线实际实现** 的 v4.0 Hub-Node 架构，不再沿用旧单体版本、旧 `/api/v1/auth/*`、旧 `/health/live`、旧 `/health/ready` 或旧 ack 风格协议叙事。

## 概览

当前主线由 **Hub 控制面** 和 **Node 本地执行面** 组成：

- `uhorse-hub`：负责 HTTP API、WebSocket 接入、任务调度、审批接口、DingTalk Stream 消息入口。
- `uhorse-node`：Node CLI 二进制入口。
- `uhorse-node-runtime`：Node 的真实运行时，负责连接循环、工作区保护、权限规则、审批请求、任务执行与结果回传。
- `uhorse-protocol`：Hub 和 Node 之间共享的命令与消息模型。
- `uhorse-config`：Hub 统一配置模型。
- `uhorse-channel`：DingTalk 等通道实现。
- `uhorse-llm`：LLM 客户端实现。

## 1. 运行时组件

### 1.1 `uhorse-hub`

Hub 是当前系统的控制面，负责：

- 暴露 HTTP API
- 接受 Node WebSocket 连接
- 保存在线节点 sender，并向节点转发任务 / 审批结果
- 调度任务并维护运行中 / 已完成任务状态
- 暴露审批查询、批准、拒绝接口
- 接入 DingTalk Stream，并把自然语言请求送入任务管道
- 在启用统一配置时初始化 `SecurityManager`、DingTalk channel 和 LLM client

### 1.2 `uhorse-node`

`uhorse-node` 只是 CLI 入口，当前主要提供：

- 正常启动 Node 运行时
- `init`：生成默认 `node.toml`
- `check`：检查工作区可用性

### 1.3 `uhorse-node-runtime`

Node 真实能力集中在 `uhorse-node-runtime`：

- 与 Hub 的连接循环和自动重连
- 注册消息携带 `auth_token`
- 心跳和状态上报
- 工作区访问约束
- git 危险命令保护
- 工作区 watcher 与新增文件自动 `git add`
- 本地权限规则执行
- 触发 `ApprovalRequest`
- 接收 `ApprovalResponse` 后继续或终止执行
- 把最终结果通过 `NodeToHub::TaskResult` 回传 Hub

### 1.4 `uhorse-protocol`

当前主线共享两层核心模型：

- `Command`
  - `File`
  - `Shell`
  - `Code`
  - `Database`
  - `Api`
  - `Browser`
  - `Skill`
- Hub / Node 双向消息

其中 `FileCommand` 当前包含 `Read`、`Write`、`Append`、`Delete`、`List`、`Search`、`Copy`、`Move`、`Info`、`CreateDir`、`Exists` 等操作。

## 2. 配置模型

## 2.1 Hub 的两种配置模式

当前 `uhorse-hub` 有两种真实配置路径。

### 统一配置

当配置文件包含以下任一段时，Hub 会按统一配置解析：

- `[server]`
- `[database]`
- `[channels]`
- `[security]`
- `[logging]`
- `[observability]`
- `[scheduler]`
- `[tools]`
- `[llm]`

统一配置当前直接驱动：

- `[server]`：Hub 监听地址和端口
- `[security]`：`jwt_secret` 等安全配置
- `[channels.dingtalk]`：DingTalk 初始化
- `[llm]`：LLM 初始化

### legacy `HubConfig`

如果配置文件不长得像统一配置，则仍按 legacy `HubConfig` 解析。

这个模式适合：

- 最小 Hub 启动
- 显式设置 `max_nodes`、`heartbeat_timeout_secs`、`task_timeout_secs`、`max_retries`
- 不需要 DingTalk / LLM

### 当前边界

统一配置 **还不会覆盖全部 Hub 专属调度字段**。像：

- `max_nodes`
- `heartbeat_timeout_secs`
- `task_timeout_secs`
- `max_retries`

当前仍会走 `HubConfig::default()`。

## 2.2 Node 配置模型

Node 当前只读取 `NodeConfig`，关键字段包括：

- `node_id`
- `name`
- `workspace_path`
- `heartbeat_interval_secs`
- `status_interval_secs`
- `max_concurrent_tasks`
- `capabilities`
- `tags`
- `git_protection_enabled`
- `watch_workspace`
- `auto_git_add_new_files`
- `require_git_repo`
- `internal_work_dir`
- `permission_rules`
- `connection.hub_url`
- `connection.reconnect_interval_secs`
- `connection.heartbeat_interval_secs`
- `connection.connect_timeout_secs`
- `connection.max_reconnect_attempts`
- `connection.auth_token`

## 3. 当前 HTTP / WebSocket 面

### 3.1 核心 Hub-Node API

当前运行时对外暴露的 Hub-Node 相关接口是：

```text
GET    /api/health
GET    /ws
GET    /api/stats
GET    /api/nodes
GET    /api/nodes/:node_id
POST   /api/nodes/:node_id/permissions
GET    /api/tasks
POST   /api/tasks
GET    /api/tasks/:task_id
POST   /api/tasks/:task_id/cancel
GET    /api/approvals
GET    /api/approvals/:request_id
POST   /api/approvals/:request_id/approve
POST   /api/approvals/:request_id/reject
POST   /api/node-auth/token
GET    /api/v1/channels/dingtalk/webhook
POST   /api/v1/channels/dingtalk/webhook
```

此外，Hub 当前还暴露运行时元数据接口：

```text
GET    /api/v1/agents
GET    /api/v1/agents/:agent_id
GET    /api/v1/skills
GET    /api/v1/skills/:skill_name
GET    /api/v1/sessions
GET    /api/v1/sessions/:session_id
GET    /api/v1/sessions/:session_id/messages
```

### 3.2 响应模型

除 `GET /api/health` 与 `GET /metrics` 外，当前 HTTP API 主要使用统一包装结构：

```json
{
  "success": true,
  "data": {},
  "error": null
}
```

`GET /api/health` 是特例，直接返回；`GET /metrics` 则返回 Prometheus 文本指标：

```json
{
  "status": "healthy",
  "version": "4.0.0-alpha.3"
}
```

### 3.3 当前已知边界

- `GET /api/tasks` 当前仍是占位实现，返回空列表。
- 查询真实任务状态应使用 `GET /api/tasks/:task_id`。
- `server.health.path` 需要与当前主线路由保持一致，推荐直接配置为 `/api/health`。

## 4. 当前协议消息模型

### 4.1 Hub → Node

当前 `HubToNode` 消息包括：

- `TaskAssignment`
- `TaskCancellation`
- `HeartbeatRequest`
- `ConfigUpdate`
- `PermissionUpdate`
- `SkillDeploy`
- `SkillRemove`
- `ApprovalResponse`
- `Disconnect`

### 4.2 Node → Hub

当前 `NodeToHub` 消息包括：

- `Register`
- `Heartbeat`
- `TaskResult`
- `TaskProgress`
- `Error`
- `ApprovalRequest`
- `Unregister`
- `SkillReport`

### 4.3 与旧文档的关键差异

当前主线 **没有** 把下面这些消息作为现行协议：

- `RegisterAck`
- `HeartbeatAck`
- `TaskAccepted`
- `TaskCompleted`
- `TaskFailed`

也就是说，当前链路是：

- Hub 下发 `TaskAssignment`
- Node 在执行期间可回传 `TaskProgress`
- Node 完成后统一回传 `TaskResult`
- 审批闭环通过 `ApprovalRequest` / `ApprovalResponse` 完成

## 5. 主链路

### 5.1 Node 引导与鉴权

当 Hub 通过统一配置启用了 `[security].jwt_secret` 后：

1. 外部可调用 `POST /api/node-auth/token`
2. Hub 为指定 `node_id` 签发 token
3. Node 把 `access_token` 放入 `connection.auth_token`
4. Node 通过 `/ws` 发起注册
5. Hub 校验 token 内 `node_id` 必须与注册 `node_id` 一致

如果 Hub 未启用 `SecurityManager`，`POST /api/node-auth/token` 会返回 `503`。

### 5.2 任务执行主链

```text
客户端 / API / DingTalk
          │
          ▼
POST /api/tasks 或 DingTalk 规划
          │
          ▼
      Hub 调度器
          │
          ▼
HubToNode::TaskAssignment
          │
          ▼
 Node 本地执行
          │
    ┌─────┴─────┐
    │           │
TaskProgress  TaskResult
    │           │
    └─────┬─────┘
          ▼
        Hub 更新任务状态
```

### 5.3 审批闭环

当前审批主链是：

1. Node 收到任务并命中本地权限规则
2. Node 生成 `NodeToHub::ApprovalRequest`
3. Hub 通过 `/api/approvals` 暴露待审批项
4. 调用 `/approve` 或 `/reject`
5. Hub 向对应 Node 发送 `HubToNode::ApprovalResponse`
6. Node 恢复执行或终止执行
7. Node 最终回传 `TaskResult`

### 5.4 Hub 重启后的 Node 自动重连

Node 连接循环内建自动重连能力，依赖：

- `connection.reconnect_interval_secs`
- `connection.max_reconnect_attempts`

当前主线已经覆盖并验证：

- Hub 重启后 Node 自动重新连接
- Node 自动重新注册
- 重连后再次提交任务仍可成功完成

### 5.5 DingTalk 与 LLM 链路

当前推荐路径是 **DingTalk Stream 模式**：

1. DingTalk 入站消息进入 Hub
2. Hub 读取统一配置里的 `[llm]`
3. 自然语言请求被规划成受本地校验约束的 `FileCommand` 或 `ShellCommand`
4. 命令通过 Hub 调度链路下发到 Node
5. 结果优先通过 `session_webhook` 原路回发
6. 如果 webhook 不可用或过期，则回退到群消息或单聊发送

兼容 webhook 路由仍保留，但不是当前推荐主路径。
