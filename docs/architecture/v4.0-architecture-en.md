# uHorse v4.0 Architecture

This document only describes the **current implemented mainline** of the v4.0 Hub-Node architecture. It no longer treats the old monolith-era flow, old `/api/v1/auth/*`, old `/health/live`, old `/health/ready`, or old ack-style protocol narrative as current.

## Overview

The current mainline is split into a **Hub control plane** and a **Node local execution plane**:

- `uhorse-hub`: HTTP API, WebSocket access, task scheduling, approval APIs, and DingTalk Stream intake.
- `uhorse-node`: CLI entrypoint for the Node binary.
- `uhorse-node-runtime`: real Node runtime for connection loops, workspace protection, permission rules, approval requests, task execution, and result return.
- `uhorse-protocol`: shared command and message model between Hub and Node.
- `uhorse-config`: unified Hub config model.
- `uhorse-channel`: channel implementations such as DingTalk.
- `uhorse-llm`: LLM client implementation.

## 1. Runtime components

### 1.1 `uhorse-hub`

Hub is the control plane of the current system. It is responsible for:

- exposing HTTP APIs
- accepting Node WebSocket connections
- keeping online node senders and forwarding tasks / approval results
- scheduling tasks and tracking running / completed state
- exposing approval list / approve / reject APIs
- receiving DingTalk Stream messages and feeding them into the task pipeline
- initializing `SecurityManager`, DingTalk channel, and LLM client when unified config is used

### 1.2 `uhorse-node`

`uhorse-node` is the CLI entrypoint. Its current responsibilities are mainly:

- starting the Node runtime
- `init`: generating a default `node.toml`
- `check`: validating workspace availability

### 1.3 `uhorse-node-runtime`

The real Node behavior lives in `uhorse-node-runtime`:

- connection loop and auto reconnect
- registration payload carrying `auth_token`
- heartbeat and status reporting
- workspace access restriction
- dangerous git command protection
- workspace watcher and auto `git add` for new files
- local permission rule evaluation
- triggering `ApprovalRequest`
- resuming or aborting execution after `ApprovalResponse`
- returning final results through `NodeToHub::TaskResult`

### 1.4 `uhorse-protocol`

The current mainline shares two core model layers:

- `Command`
  - `File`
  - `Shell`
  - `Code`
  - `Database`
  - `Api`
  - `Browser`
  - `Skill`
- bidirectional Hub / Node messages

`FileCommand` currently includes operations such as `Read`, `Write`, `Append`, `Delete`, `List`, `Search`, `Copy`, `Move`, `Info`, `CreateDir`, and `Exists`.

## 2. Configuration model

## 2.1 Two Hub config modes

`uhorse-hub` currently has two real config paths.

### Unified config

If the config file contains any of the following sections, Hub treats it as unified config:

- `[server]`
- `[database]`
- `[channels]`
- `[security]`
- `[logging]`
- `[observability]`
- `[scheduler]`
- `[tools]`
- `[llm]`

Unified config currently drives:

- `[server]`: Hub bind host and port
- `[security]`: security settings such as `jwt_secret`
- `[channels.dingtalk]`: DingTalk initialization
- `[llm]`: LLM initialization

### legacy `HubConfig`

If the config file does not look like unified config, it is still parsed as legacy `HubConfig`.

This mode is useful for:

- the smallest Hub startup path
- explicit control of `max_nodes`, `heartbeat_timeout_secs`, `task_timeout_secs`, and `max_retries`
- no DingTalk / LLM initialization

### Current boundary

Unified config **does not yet override every Hub-specific scheduler/runtime field**. Values such as:

- `max_nodes`
- `heartbeat_timeout_secs`
- `task_timeout_secs`
- `max_retries`

still fall back to `HubConfig::default()`.

## 2.2 Node config model

Node always reads `NodeConfig`. Key fields include:

- `node_id`
- `name`
- `workspace_path`
- `heartbeat_interval_secs`
- `status_interval_secs`
- `max_concurrent_tasks`
- `capabilities`
- `tags`
- `git_protection_enabled`
- `watch_workspace`
- `auto_git_add_new_files`
- `require_git_repo`
- `internal_work_dir`
- `permission_rules`
- `connection.hub_url`
- `connection.reconnect_interval_secs`
- `connection.heartbeat_interval_secs`
- `connection.connect_timeout_secs`
- `connection.max_reconnect_attempts`
- `connection.auth_token`

## 3. Current HTTP / WebSocket surface

### 3.1 Core Hub-Node APIs

The current runtime exposes these Hub-Node related endpoints:

```text
GET    /api/health
GET    /ws
GET    /api/stats
GET    /api/nodes
GET    /api/nodes/:node_id
POST   /api/nodes/:node_id/permissions
GET    /api/tasks
POST   /api/tasks
GET    /api/tasks/:task_id
POST   /api/tasks/:task_id/cancel
GET    /api/approvals
GET    /api/approvals/:request_id
POST   /api/approvals/:request_id/approve
POST   /api/approvals/:request_id/reject
POST   /api/node-auth/token
GET    /api/v1/channels/dingtalk/webhook
POST   /api/v1/channels/dingtalk/webhook
```

Hub also exposes runtime metadata endpoints today:

```text
GET    /api/v1/agents
GET    /api/v1/agents/:agent_id
GET    /api/v1/skills
GET    /api/v1/skills/:skill_name
GET    /api/v1/sessions
GET    /api/v1/sessions/:session_id
GET    /api/v1/sessions/:session_id/messages
```

### 3.2 Response model

Except for `GET /api/health` and `GET /metrics`, the current HTTP APIs mostly use a shared wrapper shape:

```json
{
  "success": true,
  "data": {},
  "error": null
}
```

`GET /api/health` is a special case and returns JSON directly; `GET /metrics` returns Prometheus text metrics:

```json
{
  "status": "healthy",
  "version": "4.0.0-alpha.3"
}
```

### 3.3 Known current boundaries

- `GET /api/tasks` is still a placeholder implementation and returns an empty list.
- Real task status should be queried through `GET /api/tasks/:task_id`.
- When `server.health.enabled = true`, `server.health.path` directly controls the Hub health route; the default value is `/api/health`.

## 4. Current protocol message model

### 4.1 Hub → Node

The current `HubToNode` variants include:

- `TaskAssignment`
- `TaskCancellation`
- `HeartbeatRequest`
- `ConfigUpdate`
- `PermissionUpdate`
- `SkillDeploy`
- `SkillRemove`
- `ApprovalResponse`
- `Disconnect`

### 4.2 Node → Hub

The current `NodeToHub` variants include:

- `Register`
- `Heartbeat`
- `TaskResult`
- `TaskProgress`
- `Error`
- `ApprovalRequest`
- `Unregister`
- `SkillReport`

### 4.3 Key differences from stale docs

The current mainline does **not** treat the following as the active protocol:

- `RegisterAck`
- `HeartbeatAck`
- `TaskAccepted`
- `TaskCompleted`
- `TaskFailed`

That means the current flow is:

- Hub sends `TaskAssignment`
- Node may emit `TaskProgress` while running
- Node finishes by returning `TaskResult`
- approval loop is driven by `ApprovalRequest` / `ApprovalResponse`

## 5. Mainline flows

### 5.1 Node bootstrap and authentication

When Hub runs from unified config with `[security].jwt_secret` enabled:

1. an external caller can request `POST /api/node-auth/token`
2. Hub issues a token for a specific `node_id`
3. Node puts the returned `access_token` into `connection.auth_token`
4. Node connects through `/ws` and sends registration
5. Hub validates that the token `node_id` matches the registering `node_id`

If Hub is running without `SecurityManager`, `POST /api/node-auth/token` returns `503`.

### 5.2 Task execution mainline

```text
client / API / DingTalk
          │
          ▼
POST /api/tasks or DingTalk planning
          │
          ▼
      Hub scheduler
          │
          ▼
HubToNode::TaskAssignment
          │
          ▼
 local Node execution
          │
    ┌─────┴─────┐
    │           │
TaskProgress  TaskResult
    │           │
    └─────┬─────┘
          ▼
      Hub updates task state
```

### 5.3 Approval loop

The current approval flow is:

1. Node receives a task and matches a local permission rule
2. Node emits `NodeToHub::ApprovalRequest`
3. Hub exposes the request via `/api/approvals`
4. caller invokes `/approve` or `/reject`
5. Hub sends `HubToNode::ApprovalResponse` to the matching Node
6. Node resumes execution or aborts it
7. Node eventually returns `TaskResult`

### 5.4 Node auto reconnect after Hub restart

The Node connection loop has built-in reconnect behavior controlled by:

- `connection.reconnect_interval_secs`
- `connection.max_reconnect_attempts`

The current mainline has already been validated for:

- automatic reconnect after Hub restart
- automatic re-registration
- successful task execution again after reconnect

### 5.5 DingTalk and LLM flow

The recommended path today is **DingTalk Stream mode**:

1. DingTalk inbound messages enter Hub
2. Hub reads `[llm]` from unified config
3. natural language is planned into a locally validated `FileCommand` or `ShellCommand`
4. the command is submitted into the Hub scheduling pipeline
5. results are returned through `session_webhook` when available
6. if the webhook is unavailable or expired, Hub falls back to group or personal message sending

Compatibility webhook routes still exist, but they are not the recommended mainline path.

## 7. Current boundaries

Do not treat the following as current v4.0 mainline facts:

- old `/health/live` / `/health/ready`
- old `/api/v1/auth/*`
- old `/api/v1/messages`
- old ack-style WebSocket protocol
- `GET /api/tasks` returning real task lists
- unified config already covering every Hub scheduler field
